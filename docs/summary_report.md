## 一、项目背景与目标

本次 AI 辅助编程与工程化实训要求独立完成一个具备实际业务价值、能跑通全流程的 Web 应用。结合本人所在学校的真实需求——校园里学生丢东西、捡东西的现象普遍，传统"贴纸质告示"或"发朋友圈"的方式信息分散、难以匹配失主——我决定做一个**面向校园场景的失物招领系统**。学校有康美、美林两个校区，因此系统需要支持多校区筛选。

项目目标不只是"能用"，而是要做到：

1. **功能闭环**：从发布 → AI 自动匹配 → 通知 → 站内信联系 → 状态置为已解决，覆盖完整业务链路；
2. **工程规范**：规范的 Git 提交、清晰的目录结构、单元测试、CI/CD、环境变量、日志系统；
3. **可复现**：别人看完 README 能在本地跑起来，能在线访问到 Demo。

## 二、个人具体贡献

整个项目由我**个人独立完成**，从前端到后端、从数据库设计到部署上线全部亲力亲为。具体包括：

### 1. 架构设计

- 设计前后端分离架构：前端 Next.js 部署在 Vercel（绑定自定义域名 jjxxyy.top），后端 Flask 部署在 PythonAnywhere，数据库用 SQLite；
- 设计 4 张核心数据表：`User`（用户）、`Item`（物品帖）、`Notification`（通知）、`Message`（站内信），并通过 `user_id`、`item_id`、`sender_id`、`receiver_id` 建立外键关联。

### 2. 后端开发（Python/Flask）

- 基于 Flask + SQLAlchemy + JWT-Extended 实现 28 个 RESTful API，覆盖用户认证、物品管理、消息聊天、通知、AI 匹配、文件上传六大模块；
- 设计 **AI 智能匹配算法**：采用**词级 Jaccard 相似度 + 字符级 Jaccard 相似度融合**，并对共同关键词加权，同时考虑同分类加成（+0.2），新帖发布自动触发匹配并向双方推送通知；
- 实现消息撤回（1 分钟内可撤回）、未读数统计、级联删除（删物品同时清消息和通知）等业务逻辑；
- 实现安全控制：未登录脱敏联系方式为 `******`、JWT 鉴权、bcrypt 密码哈希、文件扩展名白名单。

### 3. 前端开发（Next.js / TypeScript）

- 实现 11 个独立路由页面：首页、登录、注册、发布、物品详情、AI 匹配、消息、通知、个人中心、我的发布等；
- 使用 Tailwind CSS 实现响应式布局，主题色采用浅紫色（#e4cfe4）；
- 设计交互细节：导航栏未读消息红点、消息撤回时间提示、未登录点击 AI 匹配跳转登录页、性别默认头像分配、校区与分类下拉筛选。

### 4. 工程化建设

- 编写 **10 个 pytest 单元测试**覆盖注册、登录、物品发布等核心流程，使用内存数据库隔离测试；
- 配置 **GitHub Actions CI/CD**：每次推送 main 自动跑后端测试 + 前端构建；
- 引入环境变量（`.env` / `.env.example`）、Python logging 日志系统（输出到 `app.log`）；
- 撰写 [docs/prompt_log.md](./prompt_log.md) 记录 AI 辅助开发全过程，每条 prompt 配套截图，共 18 条记录。

### 5. 部署上线

- 前端部署到 Vercel，绑定自定义域名 `jjxxyy.top`；
- 后端部署到 PythonAnywhere，配置 wsgi.py 入口和静态文件映射（`/uploads/avatars/`、`/uploads/items/`、`/uploads/messages/`）；
- 整个系统已在线上跑通，从注册到完成 AI 匹配闭环全流程可访问。

## 三、遇到的最大困难及解决过程

### 困难 1：SQLite 时区丢失问题（最难，反复修复 4 次）

**问题描述**：消息撤回功能上线后，发现撤回一直失败，提示"只能在消息发出后1分钟内撤回"，但用户明明刚发完 10 秒就点了撤回。

**排查过程**：
1. 第一反应是 SQL 查询写错，检查 `Message.created_at` 的读取逻辑，发现 `datetime.now(timezone.utc)` 写入时带时区，但 SQLite 存储时把时区信息丢了，读出来变成 naive datetime；
2. 第一次尝试：读取后手动 `.replace(tzinfo=timezone.utc)` 附加时区，本地测试通过，部署后发现线上还是不对——因为 PythonAnywhere 的服务器时区与本地不同；
3. 第二次尝试：统一用 UTC 存储，所有时间戳在序列化时显式 `isoformat()`，并配合前端做相对时间显示；
4. 最终方案：写入时强制 `datetime.now(timezone.utc)`，读取后用 `.replace(tzinfo=timezone.utc)` 显式标记为 UTC，撤回时区计算用 `(datetime.now(timezone.utc) - message.created_at.replace(tzinfo=timezone.utc)).total_seconds()`，彻底解决问题。

**心得**：时区问题是后端开发的"经典坑"。SQLite 作为嵌入式数据库对时区的处理比 PostgreSQL 弱，迁移到生产环境时一定要在数据写入和读取两端显式标记时区，不能依赖数据库本身。

### 困难 2：AI 匹配中文相似度失效

**问题描述**：测试时发现，发布"寻找一卡通"和"捡到一卡通"两个相似帖子，AI 匹配的相似度居然只有 10% 左右，明显不符合预期。

**排查过程**：
1. 排查 tokenize 函数：默认按空格分词，但中文文本没有空格，分词结果就是整段字符串作为一个词；
2. 词级 Jaccard 几乎完全失效，导致相似度计算退化；
3. 解决方案：在原有词级 Jaccard 基础上**新增字符级 Jaccard 计算**——把文本拆成字符集合，用 `|字符集A ∩ 字符集B| / |字符集A ∪ 字符集B|` 计算字符相似度；
4. 最终相似度 = `(词级Jaccard + 字符级Jaccard) / 2`，对中文场景效果显著提升。

**心得**：很多算法教程默认以英文为例，但实际业务场景里中文处理需要额外考虑分词（jieba、pkuseg）或字符级方案。这次用字符级 Jaccard 是简单但有效的折中方案，未来如果数据量增大可以引入专业中文分词库。

### 困难 3：联系方式安全脱敏

**问题描述**：发现未登录用户访问 `/api/items` 时能看到所有帖子里的联系方式（微信号、手机号），隐私泄露严重。

**解决过程**：
1. 在 `get_items`、`get_item`、`search_items` 三个接口中，判断请求头是否携带 `Authorization: Bearer xxx`，未登录时返回 `contact: '******'`；
2. 同时把 AI 匹配接口加上 `@jwt_required()` 装饰器，避免被恶意调用拖垮服务器；
3. 后来在 AI Code Review 时发现这个方案有漏洞——只看 header 是否存在不看 token 是否有效（详见 [code_review.md](./code_review.md) C3），后续会用 `verify_jwt_in_request()` 真实校验。

## 四、对 AI 辅助编程的心得体会

### 1. AI 是"加速器"而非"替代品"

本次实训我全程使用 Trae（GLM-5.2）作为 AI 助手，从项目初始化、API 实现、Bug 修复到 CI/CD 配置都有 AI 参与。但实践中我深刻意识到：**AI 提供的是代码片段和建议，真正理解业务、做架构决策、判断对错的还是我自己**。比如 AI 给出的相似度算法默认按英文空格分词，我必须主动思考"中文场景怎么办"，才能发现并改进。

### 2. Prompt 质量决定 AI 输出质量

早期的 prompt 较宽泛（"帮我实现用户认证"），AI 给的代码我需要大改；后期的 prompt 越来越精确（"在 backend/app.py 第 580 行 search_items 函数里，把 LIKE 查询的 % 和 _ 字符转义，避免通配符污染搜索结果"），AI 给出的代码几乎可以直接用。**精确的定位 + 明确的诉求 + 必要的上下文**，是好 prompt 的三要素。

### 3. Prompt 日志的习惯让我受益匪浅

我要求自己每条 prompt 都记录到 [docs/prompt_log.md](./prompt_log.md) 并配截图。一开始觉得是负担，但到后期回看时发现：① 复盘 Bug 时能快速找到当时的上下文；② 写报告时素材齐全；③ 自己也养成了"先想清楚再问 AI"的习惯，避免无脑提问。

### 4. AI Code Review 价值巨大

最后阶段我请 AI 对整个项目做了一次完整的 Code Review（详见 [docs/code_review.md](./code_review.md)），发现了 45 条我之前没意识到的问题——包括严重的 XSS 漏洞、密钥硬编码、CORS 配置过宽、N+1 查询等。**这种第三方视角的审查比自审更客观**，也让我意识到"代码能跑"和"代码质量好"之间还有很大的距离。

### 5. AI 在工程化方面的提醒很关键

开发过程中 AI 多次主动提醒我："要不要加单元测试？""要不要配 CI/CD？""要不要做环境变量管理？"——这些工程化要素如果只靠自己闷头写代码很容易被忽略，但 AI 的提醒让我把项目从"功能完成"提升到"工程规范"的层面，恰好契合本次实训考核对工程规范的重视。

## 五、个人反思与未来改进

### 反思

1. **前期对安全的重视不够**：未登录脱敏、密钥管理这些问题如果在设计阶段就考虑，能省掉后期返工；
2. **测试覆盖率偏低**：10 个测试只覆盖了核心 API，AI 匹配、消息撤回、文件上传等关键功能没有测试，CI 上的绿色勾并不能保证这些功能真的对；
3. **前端代码复用差**：聊天 UI 在 messages 和 items/[id] 两个页面几乎复制了一份，后续任何修改都要改两处，违反 DRY 原则。

### 未来改进方向

1. **引入 Flask-Migrate 数据库迁移**，避免后续改表困难；
2. **抽取统一的前端 API 客户端**（`apiFetch` 函数自动注入 token、统一错误处理、401 拦截）；
3. **抽取共享 ChatPanel 组件**，messages 和 items/[id] 页面复用；
4. **用 React Context 替代每页 validateToken**，减少重复请求；
5. **接入 Flask-Limiter 限流**，防止 AI 匹配等重计算接口被滥用；
6. **把 JWT 从 localStorage 迁移到 httpOnly cookie**，提升安全性。

## 六、总结

本次实训让我完整走通了一个 Web 项目从 0 到 1 的全过程——从需求分析、架构设计，到编码实现、测试、部署、上线，7 天的高强度开发让我深刻体会到，"代码能跑起来"只是起点，"代码能跑得稳、跑得久、别人能接手"才是工程化的终点。AI 辅助编程让编码效率大幅提升，很多接口 30 分钟就能完成，但这也让我更加清醒地认识到：架构思考、业务理解、质量把控这些"人的工作"反而比以往更重要了——因为 AI 帮你写得快，所以你能更快地写出更多需要审视和优化的代码，如果没有 Code Review 和测试的意识，AI 带来的速度优势很快就会被技术债的累积抵消。回顾整个开发过程，我也看到了自己明显的短板：前期安全设计意识不足，密钥硬编码、联系方式伪造脱敏、CORS 配置过宽等问题都是后期才暴露的；测试驱动开发的习惯尚未建立，代码写完后才补测试，导致 AI 匹配、消息撤回等关键逻辑至今缺少自动化测试覆盖；前端代码复用意识薄弱，ChatPanel 在两个页面几乎逐行复制，维护成本翻倍；对 SQLite 生产环境局限性认识不足，时区丢失问题反复修复了 4 次才彻底解决。这些教训让我更加清醒地认识到，独立完成一个能跑的项目只是第一步，写出高质量、可维护、安全可靠的系统才是真正的挑战。但正是这些踩过的坑，让我建立了两项真正有价值的能力——"工程化思维"让我在写代码时能预见问题，"AI 协作思维"让我在遇到问题时能高效借助工具解决。这两项能力，将是我后续学习和工作中最重要的底层资产，也让我明白，AI 能加速实现，但真正的成长，永远来自对每一次失误的反思和对每一次成功的复盘。
